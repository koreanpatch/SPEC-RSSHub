---
name: build-context
description: 'Generate .planning/codebase/ reference snapshots for Sunbi. Run once before a new work session or after major structural changes. All other skills read from these instead of scanning src/. Usage: /build-context'
---

<objective>
Build lightweight reference snapshots of the Sunbi codebase into .planning/codebase/
so that all other skills (feature, audit-pass, do) can read 2KB summaries instead
of full source files. Massive token savings per session.
</objective>

<process>

## 1. entrypoints.md

Read all 4 entrypoints:

- src/entrypoints/background.ts
- src/entrypoints/content.ts
- src/entrypoints/sidepanel/main.tsx + App.tsx
- src/entrypoints/get-started/

Write .planning/codebase/entrypoints.md:

- One paragraph per entrypoint: what it does, what it owns, what chrome APIs it uses
- The isolation table (what can import what)
- All message types handled in background.ts (name + payload shape, one line each)

## 2. store-shape.md

Read:

- src/entrypoints/sidepanel/store.ts
- src/entrypoints/sidepanel/store/ (all files)

Write .planning/codebase/store-shape.md:

- Full state shape as a TypeScript interface block (no implementation, just types)
- Every action name + signature + one-line description
- Which fields are persisted to chrome.storage
- Which store (appStore vs dictionaryStore) owns what

## 3. routing.md

Read App.tsx and any routing config.

Write .planning/codebase/routing.md:

- VALID_VIEW_IDS array (copy exactly)
- ViewId union type (copy exactly)
- NAV_ITEMS array (copy exactly)
- renderView switch cases listed (viewId → component)
- Rule: when adding a new view, all 4 of these must be updated

## 4. i18n.md

Find and read the i18n strings file (src/shared/i18n/ or strings.ts).

Write .planning/codebase/i18n.md:

- Total string count
- Namespace structure (if any)
- Pattern for adding new strings (the t() call + where to add the key)
- Any Korean-specific formatting rules in the codebase

## 5. shared-api.md

Read src/shared/ directory.

Write .planning/codebase/shared-api.md:

- Every exported utility/helper (name + what it does, one line)
- Storage helpers (keys, types, defaults)
- Config constants
- Type definitions exported from shared

## After all files written

Add a header to each file:

<!-- Generated by /build-context on YYYY-MM-DD. Re-run after major structural changes. -->

Commit: "chore: build .planning/codebase/ context snapshots"

Report: how many tokens each file is (rough estimate via wc -c), and total snapshot size vs what the raw source files would cost to read.

</process>

<rules>
- Snapshots are summaries, not copies — omit implementation details
- The goal is: another Claude session can understand the full codebase shape from these 5 files alone
- If a source file hasn't changed since last snapshot (check git log), skip it
- Never put secrets or .env values in snapshots
</rules>
